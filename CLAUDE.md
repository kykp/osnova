# Osnova — правила проекта

> Коробочная CMS на Next 15 + Payload 3. Завуч/администратор школы должен установить и запустить за ~15 минут.

## Стек

- **Next.js 16** (App Router) — фронт сайта.
- **Payload 3** (встроенный в тот же Next-проект, route group `(payload)`) — админка/CMS.
- **PostgreSQL** — единая БД.
- **TypeScript** — везде, `strict: true`.
- **pnpm** — единственный пакетный менеджер. Не npm, не yarn.
- **Docker Compose** — для подъёма Postgres локально и в проде (см. `docker-compose.yml`).
- **Zustand** — если в клиенте нужен глобальный стейт. Не Redux/Jotai/MobX.

## Архитектура

**Монолит.** Один процесс Next.js обслуживает и сайт, и `/admin`. Никаких микросервисов, выделенных API-серверов, очередей, брокеров. «Сторонний бэкендер» в нашем понимании — это человек, расширяющий монолит (новые коллекции/хуки Payload), а не автор внешнего сервиса.

Структура (после установки create-payload-app):

```
.
├── app/                          # само приложение Next + Payload
│   ├── src/
│   │   ├── app/(payload)/        # админка /admin и payload API
│   │   ├── app/(frontend)/       # публичный сайт
│   │   ├── collections/          # коллекции Payload (Users, Media, ...)
│   │   └── payload.config.ts
│   ├── next.config.ts
│   ├── tsconfig.json
│   ├── package.json
│   └── .env.example              # DATABASE_URI, PAYLOAD_SECRET, NEXT_PUBLIC_SERVER_URL
├── docker-compose.yml            # Postgres
├── .env.example                  # POSTGRES_USER/PASSWORD/DB
├── CLAUDE.md
└── TODO.md
```

## Стили

- **CSS Modules** (`*.module.css` рядом с компонентом). Никакого БЭМ в `globals.css`, никакого Tailwind, никакого CSS-in-JS.
- В `globals.css` — только токены (CSS-переменные) и reset.
- Переменные — единственный источник цветов/отступов/шрифтов. Не вкомпиливать литералы.

## Принципы

### Коробочный продукт, не SaaS
Завуч ставит сам за 15 минут. Минимум настроек, дефолты вместо рычагов. Каждая новая опция — повод спросить «а не убрать ли её и поставить разумный дефолт?».

### Scalability first (через полгода / сторонний фронт / сторонний бэк)
Любое решение прогоняется по трём аудиториям:
1. **Мы через полгода** — поймём ли сами, что тут происходит, без контекста?
2. **Сторонний фронтендер** — поймёт ли структуру без объяснений?
3. **Сторонний бэкендер** — сможет ли расширить (новая коллекция, хук, плагин) без переписывания?

Из этого следует: **токены, а не литералы; стабильные контракты типов; явные имена; никаких магических констант.**

### Только русскоязычные клиенты
Тексты UI/админки/писем — по-русски (или с возможностью русифицировать через i18n Payload). Английский только в коде/коммитах/идентификаторах.

### Никакой ИИ-атрибуции
В коде, комментариях, именах файлов, сообщениях коммитов — **ни слова** про ИИ-агента, Claude, Copilot и т.п. `Co-Authored-By` не добавлять.

## Code style

- **Без лишних комментариев.** Хорошие имена документируют сами себя. Комментарий — только когда «почему» неочевидно (скрытое ограничение, инвариант, обход бага).
- **Не оборачивать в try/catch на всякий случай.** Доверяй внутренним вызовам, валидируй только на границах системы (вход пользователя, внешние API).
- **Не плодить абстракции вперёд требований.** Три похожих строки лучше преждевременной обобщённой функции.

## Команды

```bash
# из корня — Postgres
docker compose up -d

# из app/ — приложение
cd app
cp .env.example .env        # один раз; затем openssl rand -hex 32 → PAYLOAD_SECRET
pnpm install
pnpm dev                    # http://localhost:3000, /admin для админки
```
