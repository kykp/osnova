# Зацепки в потроха Payload

> Файл фиксирует все места, где наш код полагается на **внутренние детали Payload** (CSS-классы, DOM-структура, форматы id). Эти места **не покрыты публичным API** — при апгрейде Payload могут поменяться, и тогда нужно поправить здесь.
>
> **Правило проекта:** любая новая кастомизация админки, цепляющаяся за внутренние классы/DOM/id Payload, **добавляется в этот файл** (или в аналогичный рядом со своим компонентом). Без этого через год никто не вспомнит, что и где у нас прибито гвоздями.

Версия Payload, на которую опирались при разработке: **3.85.1** (см. `app/package.json`).

---

## Что использует `BlockVisibilityMenu`

### Публичный API `@payloadcms/ui` (стабильное, ломается громко через TS)

| Импорт | Зачем |
|---|---|
| `useField` | Чтение/запись `layout.N.enabled` в форме |
| `useAllFormFields` | Перебор всех `layout.N.enabled` для подсветки скрытых строк |

### CSS-классы Payload (хрупкое, может молча отвалиться)

| Класс | Где используется | Что отвалится |
|---|---|---|
| `.array-actions__button` | `onMouseDown` ловит клик по «...»-кнопке строки → запоминает индекс строки | Не сможем определить, какая строка открыла popup |
| `.popup__content` / `.popup__hidden-content` | MutationObserver ловит появление/скрытие popup-меню | Не появятся наши пункты в меню |
| `.popup-button-list` | Контейнер списка кнопок внутри popup — целевой узел для React-портала | Не появятся наши пункты в меню |
| `.array-actions__action` | Стилевой класс пункта меню Payload (мы добавляем такой же на нашу кнопку «Скрыть/Показать») | Наша кнопка будет без стиля |
| `.collapsible__toggle-wrap` | Фон + рамка скрытого блока (штриховка) | Пропадёт серая штриховка скрытого блока |
| `.blocks-field__block-pill` | Перечёркиваем при `enabled=false` | Пропадёт перечёркивание |
| `.blocks-field__block-number` | Перечёркиваем номер блока при `enabled=false` | Пропадёт перечёркивание номера |
| `.blocks-field__block-header` | Псевдоэлемент `::after` с подписью «— скрыт на сайте» | Пропадёт подпись |
| `.collapsible__content` | Полупрозрачное содержимое скрытого блока (если развёрнут) | Содержимое не будет тускнеть |

### DOM-структура и id

| Что | Где | Что отвалится |
|---|---|---|
| Popup рендерится через `createPortal(..., document.body)` | Faceless-UI Popup в `@payloadcms/ui` | Если переедет в другое место — MutationObserver не найдёт его (наблюдаем за `document.body`) |
| Каждая строка блока имеет id вида `layout-row-N` | Генерится Payload в `BlockRow` (`${parentPath.split(".").join("-")}-row-${rowIndex}`) | Не сможем сопоставить popup со строкой (`requestDelete` не узнает индекс); подсветка скрытого блока не сработает |

## Где это всё в коде

- `BlockVisibilityMenu.tsx` — логика
- `BlockVisibilityMenu.module.css` — стили и `display: none` на оригинальный «Удалить»
- Все 18 блоков в `app/src/blocks/*/config.ts` — импортируют `enabledField`
- `app/src/blocks/_shared/enabledField.ts` — определение поля
- `app/src/blocks/RenderBlocks.tsx` — фильтр `enabled === false` на сайте
- `app/src/collections/Pages.ts` — подключает компонент через `ui`-поле `blockVisibilityMenu`

---

## Что НЕ зависит от внутренних Payload (переживёт любой мажор)

- Поле `enabled: boolean` в каждом блоке (наша декларация)
- Фильтр блоков в `RenderBlocks.tsx` (наш React-код)
- Хранение значения `enabled` в БД (стандартное поле Payload в JSONB колонке `layout`)

То есть **данные** в безопасности всегда. Ломается только **визуальная обёртка** в админке.

---

## Как чинить при апгрейде Payload

1. После `pnpm update payload @payloadcms/*` проверить в админке: открывается ли popup-меню блока, есть ли в нём наши пункты, работает ли подсветка скрытого блока.
2. Если что-то пропало — открыть DevTools на странице редактирования, посмотреть актуальные классы у popup и строки блока.
3. Сверить с таблицей выше, обновить селекторы в `BlockVisibilityMenu.tsx` / `.module.css`.
4. Обновить версию Payload в начале этого файла.
